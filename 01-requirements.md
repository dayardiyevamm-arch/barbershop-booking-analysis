# 1. Требования к продукту

## 1.1. Бизнес-контекст

Сеть барбершопов в Астане обслуживает ~1500 клиентов в месяц. Сейчас запись идёт по телефону и через мессенджеры: администратор вручную ведёт расписание в Excel.

**Проблемы текущего процесса:**
- 15–20% записей теряются или дублируются из-за ручного ведения.
- Клиент не видит, у какого мастера есть свободное время.
- Невозможно быстро перенести запись — нужно звонить.
- Нет статистики по загрузке мастеров и отменам.

**Цель продукта:** снизить долю потерянных записей до <3% и дать клиенту самостоятельно управлять визитом через приложение.

## 1.2. Метрики успеха

| Метрика | Текущее значение | Целевое (через 6 мес.) |
|---|---|---|
| Доля онлайн-записей от общего числа | 0% | 70% |
| Доля потерянных/дублирующихся записей | 15–20% | <3% |
| Среднее время оформления записи клиентом | 3–5 мин (звонок) | <1 мин |
| Доля отмен позднее чем за 2 часа | не отслеживается | <10% |

## 1.3. Роли пользователей

| Роль | Описание | Основные действия в MVP |
|---|---|---|
| **Клиент** | Физлицо, записывающееся на услугу | Просмотр слотов, создание/отмена/перенос записи |
| **Мастер** | Сотрудник барбершопа | Просмотр своего расписания (вне скоупа данного ТЗ) |
| **Администратор** | Управляющий филиалом | Управление мастерами/услугами (вне скоупа данного ТЗ) |

> **Скоуп текущего ТЗ:** только клиентская часть — бронирование, отмена, перенос.

## 1.4. Глоссарий

| Термин | Определение |
|---|---|
| **Слот** | Интервал времени фиксированной длины (например, 30 мин), на который можно записать клиента к мастеру. |
| **Запись (Booking)** | Забронированный клиентом слот с привязкой к услуге и мастеру. |
| **Услуга (Service)** | Тип работы: стрижка, оформление бороды и т.д. У каждой услуги есть длительность и цена. |
| **Филиал (Branch)** | Физическое расположение барбершопа. |
| **Окно отмены** | Минимальное время до начала визита, в которое клиент ещё может бесплатно отменить/перенести запись. По умолчанию — 2 часа. |

## 1.5. Функциональные требования

### FR-1. Просмотр свободных слотов
Клиент должен иметь возможность посмотреть список свободных слотов на выбранную дату, с фильтрами по:
- филиалу,
- мастеру (опционально — «любой свободный»),
- услуге (длительность слота зависит от услуги).

### FR-2. Создание записи
Клиент должен иметь возможность забронировать свободный слот, указав:
- услугу,
- мастера,
- дату и время начала,
- контактный номер (подставляется из профиля).

Система должна:
- проверить, что слот всё ещё свободен,
- создать запись со статусом `CONFIRMED`,
- вернуть подтверждение с ID записи.

### FR-3. Отмена записи
Клиент должен иметь возможность отменить свою запись **не позднее чем за 2 часа** до начала. После отмены слот должен освободиться.

**Правила:**
- Отменить можно только свою запись.
- Отменить можно только запись в статусе `CONFIRMED`.
- После окна отмены (<2 ч до начала) — отмена невозможна.

### FR-4. Перенос записи
Клиент должен иметь возможность перенести свою запись на другой свободный слот **не позднее чем за 2 часа** до начала.

**Правила:**
- Переносить можно только свою запись в статусе `CONFIRMED`.
- Новый слот должен быть свободен.
- Услуга и мастер могут меняться при переносе.
- История переносов сохраняется (для аналитики).

### FR-5. Просмотр своих записей
Клиент должен иметь возможность посмотреть список своих записей с фильтрами:
- предстоящие,
- прошедшие,
- отменённые.

## 1.6. Нефункциональные требования

| Категория | Требование |
|---|---|
| **Производительность** | Время ответа API на GET-запросы (просмотр слотов, списка записей) — не более 500 мс в 95-м перцентиле. |
| **Доступность** | Uptime сервиса — не менее 99.5% в месяц (≈3.6 часа простоя). |
| **Масштабируемость** | Система должна выдерживать 500 одновременных пользователей и 50 запросов в секунду. |
| **Безопасность** | Все запросы — через HTTPS. Авторизация — JWT-токен. Клиент может видеть и менять только свои записи. |
| **Локализация** | Интерфейс и сообщения об ошибках — на русском, казахском и английском. |
| **Часовой пояс** | Все даты/время хранятся в UTC, отображаются в часовом поясе филиала (Asia/Almaty). |
| **Логирование** | Все действия с записями (создание, отмена, перенос) логируются с user_id, timestamp и результатом. |

## 1.7. Ограничения и допущения

**Ограничения:**
- Оплата услуг происходит офлайн (в кассе), онлайн-платежи — вне скоупа MVP.
- Пуш-уведомления и SMS — вне скоупа этого ТЗ (описаны отдельным документом).
- Клиент не может отменить запись, если она уже в статусе `COMPLETED` или `CANCELLED`.

**Допущения:**
- Каждый клиент имеет уникальный номер телефона, по которому авторизуется.
- Длительность слота = длительность услуги (округляется вверх до 30 мин).
- Один клиент не может иметь две активные записи на одно и то же время.
