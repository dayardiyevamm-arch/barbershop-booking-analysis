# 7. Тестирование

Документ описывает подход к тестированию фичи бронирования и базовый чек-лист тест-кейсов.

## 7.1. Виды тестирования

| Уровень | Что проверяет | Кто делает | Инструменты |
|---|---|---|---|
| **Unit-тесты** | Отдельные функции: расчёт `end_at`, валидация окна отмены, проверка пересечений | Разработчики | JUnit, pytest, Jest |
| **Интеграционные** | Booking Service + БД: корректность SQL-запросов, транзакций, блокировок | Разработчики | Testcontainers, in-memory БД |
| **API-тесты** | Эндпоинты целиком: статусы, структура ответов, коды ошибок | QA-инженер / аналитик | Postman, Swagger, REST Assured |
| **End-to-End (E2E)** | Полный сценарий от UI до БД | QA-инженер | Cypress, Playwright |
| **Нагрузочные** | 500 одновр. пользователей, 50 RPS, p95 < 500 мс | QA / DevOps | k6, JMeter, Locust |
| **Регрессионные** | Что старая функциональность не сломалась после изменений | QA | Автотесты в CI/CD |
| **UAT (приёмочные)** | Соответствие acceptance criteria | PO / аналитик | Ручная проверка по сценариям |

## 7.2. Тест-дизайн: техники

Для покрытия сценариев использованы:

- **Эквивалентные классы** — группируем входные значения (например, «до визита >2 часов» / «<2 часов»).
- **Граничные значения** — особое внимание к границе: ровно 2 часа, 1 час 59 минут, 2 часа 1 минута.
- **Таблица решений** — для сложной логики переноса (см. ниже).
- **State transition testing** — переходы между статусами записи.

## 7.3. Таблица решений: отмена записи

| Условие | К1 | К2 | К3 | К4 | К5 |
|---|---|---|---|---|---|
| Запись существует | ✓ | ✓ | ✓ | ✓ | ✗ |
| Запись принадлежит клиенту | ✓ | ✓ | ✓ | ✗ | — |
| Статус `CONFIRMED` | ✓ | ✓ | ✗ | — | — |
| До визита ≥ 2 часов | ✓ | ✗ | — | — | — |
| **Ожидаемый результат** | 200 OK | 422 WINDOW_EXPIRED | 422 INVALID_STATE | 403 FORBIDDEN | 404 NOT_FOUND |

## 7.4. Чек-лист тест-кейсов

### 🔹 Получение слотов (GET /slots)
- [ ] Возврат корректного списка свободных слотов на выбранную дату.
- [ ] Слоты, уже занятые другими записями, не возвращаются.
- [ ] При `master_id = null` возвращаются слоты всех активных мастеров филиала.
- [ ] Фильтр по конкретному мастеру работает корректно.
- [ ] Длительность слота = длительности услуги.
- [ ] При запросе на прошедшую дату возвращается пустой список или 400.
- [ ] Слоты за пределами рабочего времени филиала не возвращаются.

### 🔹 Создание записи (POST /bookings)
- [ ] Успешное создание при валидных данных (201).
- [ ] Запись появляется в БД со статусом `CONFIRMED`.
- [ ] Запись в `booking_history` с `event_type = CREATED`.
- [ ] Уведомление отправлено.
- [ ] 409 при попытке забронировать уже занятый слот.
- [ ] 409 при пересечении со своей существующей записью.
- [ ] 422 при `start_at` в прошлом.
- [ ] 422 при слоте вне рабочих часов.
- [ ] 422 если мастер не оказывает эту услугу.
- [ ] 400 при отсутствии обязательного поля.
- [ ] 401 без токена / с истёкшим токеном.
- [ ] **Race condition:** при 2 параллельных POST на один слот — только одна запись создаётся, вторая получает 409.

### 🔹 Отмена (PATCH /bookings/{id}/cancel)
- [ ] Успешная отмена при до визита > 2 часов (200).
- [ ] Статус меняется на `CANCELLED`.
- [ ] Слот снова появляется в свободных.
- [ ] Запись в `booking_history` с `event_type = CANCELLED`.
- [ ] 422 за 1ч 59 мин до визита (`WINDOW_EXPIRED`).
- [ ] 200 ровно за 2 часа до визита (граничное значение).
- [ ] 422 при повторной отмене уже отменённой записи.
- [ ] 422 при попытке отменить `COMPLETED` запись.
- [ ] 403 при попытке отменить чужую запись.
- [ ] 404 при несуществующем `id`.

### 🔹 Перенос (PATCH /bookings/{id}/reschedule)
- [ ] Успешный перенос на свободный слот (200).
- [ ] `booking.id` не меняется.
- [ ] Старый слот освобождается, новый занимается.
- [ ] Запись в `booking_history` с `event_type = RESCHEDULED` и `old_value/new_value`.
- [ ] Можно менять мастера.
- [ ] Можно менять услугу.
- [ ] 409 при попытке перенести на занятый слот.
- [ ] 422 при переносе за < 2 часа до визита.
- [ ] 422 при переносе в прошлое.
- [ ] Если `master_id` / `service_id` не переданы — остаются прежними.
- [ ] **Race condition:** 2 клиента переносят на один слот — один получает 200, другой 409.

### 🔹 Список записей (GET /bookings)
- [ ] Фильтр `status=upcoming` возвращает только `CONFIRMED` с `start_at > NOW()`.
- [ ] Фильтр `status=past` возвращает `COMPLETED` и `NO_SHOW`.
- [ ] Фильтр `status=cancelled` возвращает только `CANCELLED`.
- [ ] Сортировка по `start_at` ASC для upcoming, DESC для остальных.
- [ ] Пагинация работает корректно (`limit`, `offset`).
- [ ] Клиент видит только свои записи.

### 🔹 Безопасность
- [ ] Все эндпоинты требуют JWT.
- [ ] Клиент не может получить/изменить чужую запись (проверка `client_id` на бэкенде, не только во фронтенде).
- [ ] SQL-инъекции невозможны (проверка на специальных payload'ах).
- [ ] Нет утечки чувствительных данных в ошибках.

### 🔹 Производительность
- [ ] p95 времени ответа GET /slots < 500 мс при 50 RPS.
- [ ] p95 POST /bookings < 800 мс при 50 RPS.
- [ ] Нет deadlock'ов при 100 параллельных бронированиях.

## 7.5. Тест-данные

Минимальный набор для ручного и автоматизированного тестирования:

- 1 филиал (10:00–22:00, Asia/Almaty).
- 3 мастера с разным расписанием и разным набором услуг.
- 3 услуги: стрижка (30 мин), борода (30 мин), стрижка+борода (60 мин).
- 3 клиента: один с активными записями, один без записей, один с отменёнными записями.

## 7.6. Критерии выхода на прод (Exit Criteria)

Фича готова к релизу, если:
- 100% acceptance criteria из [02-user-stories.md](02-user-stories.md) покрыты автотестами и проходят.
- Регрессионные тесты зелёные.
- Нет открытых дефектов уровня Critical или High.
- Нагрузочный тест пройден (p95 в пределах NFR).
- Документация API актуальна.
- PO подтвердил UAT.
